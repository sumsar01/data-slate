## Executive Summary

### What is Data-Slate?
A 40K-themed campaign management system that:
1. Records audio sessions
2. Automatically transcribes them
3. Rewrites transcripts with Warhammer 40K terminology (Groq LLM)
4. Extracts named entities (NPCs, Locations, Factions, Items)
5. Builds a searchable wiki
6. Groups notes into "sessions" (campaigns)
7. Generates session summaries
8. Shares read-only access via tokens

### Tech Stack
- **Frontend:** React 19 + Vite + React Router
- **Backend:** Hono (Bun runtime)
- **Database:** Turso (SQLite-compatible)
- **Storage:** Cloudflare R2 (S3 API)
- **LLM:** Groq SDK (speech-to-text, NER, text generation)
- **Deploy:** Vercel (frontend), Railway (backend)

### Key Data Structures

```typescript
// Campaign grouping
type DateGroup = {
  date: string                    // YYYY-MM-DD
  session_id: string | null       // UUID
  session_name: string | null     // User-friendly name
  session_summary: string | null  // Groq-generated
  notes: Note[]                   // All notes for date
}

// Individual recording
type Note = {
  id: string
  date: string
  title: string                   // Generated by Groq
  transcript: string              // Flavoured with 40K terms
  audio_url: string | null        // R2 public URL
  duration_s: number
  tags: Tag[]                     // NPC, Location, Faction, etc.
  entities: Entity[]              // Extracted by Groq
  created_at: string
}

// Named entity
type Entity = {
  name: string
  type: EntityType               // NPC | Location | Faction | Item | Other
}
```

### Database Tables
1. **notes** - Audio recordings + transcriptions + extracted entities
2. **session_overrides** - Campaign groupings + summaries
3. **shares** - Read-only share tokens + expiry
4. **entities** - Wiki entries + deduplication + images + dossiers

### Frontend Routes
| Path | Purpose |
|------|---------|
| `/` | Main log view (2-panel: notes left, detail right) |
| `/record` | Audio recorder |
| `/admin-mechanicus` | Admin panel (export, shares, entity sync) |
| `/wiki` | Entity browser |
| `/wiki/:id` | Entity detail + mentions |
| `/share/:token` | Public read-only session |

### API Endpoints (29 total)
- **Notes:** POST (upload+transcribe), GET (list/detail), DELETE
- **Dates:** GET (aggregate by date + sessions)
- **Sessions:** POST (create), PATCH (update), POST summary
- **Shares:** POST (create), GET (list), DELETE (revoke), GET shared/:token (public)
- **Wiki:** GET (list/detail), POST/PATCH (CRUD), POST sync, POST merge

## Key Insights

### Architecture Patterns
1. **Monorepo with Bun workspaces** - Three apps: web, api, recorder
2. **REST API** - Hono routes, no GraphQL
3. **Polling architecture** - Frontend polls `/dates` every 30 seconds (not real-time)
4. **Fire-and-forget async** - Entity extraction doesn't block note creation
5. **JSON in SQLite** - tags, entities, dates stored as JSON strings
6. **Type sync challenge** - Types in 3 places (shared pkg, api lib, web shared) kept manually in sync

### Missing Pieces
- No tests (AGENTS.md notes "no tests exist yet")
- No migrations (schema defined inline in routes with CREATE TABLE IF NOT EXISTS)
- No authentication/authorization
- No real-time updates (polling instead)
- No session/campaign "model" per se (just SessionOverride naming override)

### Code Quality
- Clean TypeScript (strict mode implied)
- React hooks over Redux (simple enough)
- CSS co-located with components
- Flat ESLint config (v10)
- No Prettier
- Terminal/80s aesthetic (scanlines, monospace fonts)

### Deployment
- **Frontend:** Auto-deploys main → Vercel
- **Backend:** Manual `railway up` from apps/api/
- **Database:** Turso cloud (no local setup)
- **Storage:** Cloudflare R2 (no local file handling)

## Common Development Tasks

### Add a New Campaign-Like Feature
1. Add API endpoint in `/apps/api/src/routes/`
2. Add table to Turso via CREATE TABLE IF NOT EXISTS
3. Add frontend page in `/apps/web/src/pages/`
4. Add component in `/apps/web/src/components/` if needed
5. Add route in `/apps/web/src/main.tsx`
6. Update types in all 3 type definition locations

### Extend Entity Wiki
- Edit `/apps/api/src/routes/wiki.ts` for backend logic
- Edit `/apps/web/src/pages/WikiPage.tsx` for display
- New entity types go in EntityType union in shared types

### Add Session/Campaign Features
- Core session logic: `/apps/api/src/routes/sessions.ts`
- Frontend UI: `/apps/web/src/components/SessionOverride.tsx` + NoteList
- Session state flows through DateGroup type

### Debug Data Flow
1. Check `/apps/api/src/routes/notes.ts` for note storage
2. Check `/apps/api/src/routes/dates.ts` for aggregation
3. Check `/apps/web/src/hooks/useDateGroups.ts` for polling
4. Check `/apps/web/src/data/api.ts` for API calls
